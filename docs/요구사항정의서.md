# 요구사항 정의서 — 학습자료 기반 스터디 챗봇

> 사용자가 업로드한 학습자료(PDF 등)를 바탕으로 질문에 답하고, 퀴즈를 자동 생성해주는 RAG 기반 스터디 챗봇 서비스.

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 서비스명 | 학습자료 기반 스터디 챗봇 (가칭) |
| 한 줄 정의 | 내가 올린 강의자료·PDF를 학습해서 질문에 답하고 퀴즈를 만들어주는 챗봇 |
| 핵심 기술 | RAG · WebSocket · Redis |
| 백엔드 | Spring Boot 3.x + Spring AI 1.x (별도 서버 직접 설계) |
| 프론트엔드 | React |
| 데이터 저장 | PostgreSQL(일반 데이터) · Redis(벡터·세션·캐시) |
| 인증 방식 | JWT + 리프레시 토큰 (소셜 로그인은 확장) |

---

## 2. 기능 요구사항

기능은 **핵심(MVP)**과 **확장(이후 추가)**으로 구분한다.

### 2.1 회원 / 인증 — 핵심

| ID | 요구사항 |
|----|----------|
| AUTH-1 | 이메일·비밀번호로 회원가입한다 |
| AUTH-2 | 로그인하면 JWT 액세스 토큰을 발급받는다 |
| AUTH-3 | 액세스 토큰이 만료되면 리프레시 토큰으로 재발급받는다 |
| AUTH-4 | 로그아웃하면 토큰이 무효화된다 |
| AUTH-5 | 회원가입 시 이메일 형식·비밀번호 규칙을 검증한다 |

### 2.2 소셜 로그인 — 확장

| ID | 요구사항 |
|----|----------|
| SOCIAL-1 | 구글 OAuth 2.0으로 로그인·가입한다 |
| SOCIAL-2 | 카카오 OAuth 2.0으로 로그인·가입한다 |
| SOCIAL-3 | 소셜 계정과 자체 계정을 일관된 구조로 관리한다 |

### 2.3 학습자료 관리 — 핵심

| ID | 요구사항 |
|----|----------|
| DOC-1 | 학습자료(PDF 등)를 업로드한다 |
| DOC-2 | 업로드한 자료에서 텍스트를 추출한다 |
| DOC-3 | 추출한 텍스트를 검색 가능한 형태로 저장한다 (임베딩 → 벡터 저장) |
| DOC-4 | 자료의 처리 상태(처리 중 / 완료 / 실패)를 사용자에게 표시한다 |
| DOC-5 | 내가 올린 자료 목록을 조회한다 |
| DOC-6 | 자료를 삭제한다 |

### 2.4 챗봇 (RAG) — 핵심

| ID | 요구사항 |
|----|----------|
| CHAT-1 | 내 자료에 대해 질문한다 |
| CHAT-2 | 답변은 자료 내용을 근거로 생성된다 |
| CHAT-3 | 답변이 실시간으로 한 글자씩 표시된다 (WebSocket 스트리밍) |
| CHAT-4 | 답변의 근거가 된 자료 위치를 함께 보여준다 |
| CHAT-5 | 자료가 없거나 답을 찾을 수 없으면 안내 메시지를 보여준다 |

### 2.5 퀴즈 — 확장

| ID | 요구사항 |
|----|----------|
| QUIZ-1 | 자료를 기반으로 퀴즈를 자동 생성한다 |
| QUIZ-2 | 퀴즈를 풀고 채점받는다 |
| QUIZ-3 | 오답을 기록하고 조회한다 |

### 2.6 대화 관리 — 확장

| ID | 요구사항 |
|----|----------|
| CONV-1 | 이전 대화 기록을 저장하고 조회한다 |

## 3. 비기능 요구사항

| ID | 분류 | 요구사항 |
|----|------|----------|
| NFR-1 | 보안 | 비밀번호는 해싱하여 저장한다 (평문 저장 금지) |
| NFR-2 | 보안 | 토큰을 안전하게 관리한다 (리프레시 토큰은 Redis 저장) |
| NFR-3 | 보안 | 시크릿(API 키·DB 비밀번호 등)은 코드·저장소에 포함하지 않는다 |
| NFR-4 | 제약 | 업로드 파일 형식은 PDF로 한다 (추후 확장 가능) |
| NFR-5 | 제약 | 업로드 파일 최대 용량 제한을 둔다 *(구체적 수치는 개발 단계에서 결정 — TBD)* |
| NFR-6 | 요청 제한 | 한 사용자가 분당 일정 횟수 이상 질문할 수 없다 (Redis 기반 rate limiting, LLM 비용 보호) |
| NFR-7 | 성능 | 같은 질문은 캐싱하여 빠르고 저렴하게 응답한다 (Redis) |
| NFR-8 | 안정성 | 오류 응답 형식을 일관되게 통일한다 |
| NFR-9 | 확장성 | 서버가 세션 상태를 저장하지 않는 구조(JWT)로 설계한다 |

## 4. 개발 우선순위 (워킹 스켈레톤)

기능을 한 번에 다 만들지 않고, 작동하는 최소 단위부터 단계적으로 확장한다.

### 1단계 — 뼈대 (가장 먼저, 끝까지 관통)
- 자체 회원가입 · 로그인 (JWT)
- 자료 업로드 (텍스트 기반, 단순)
- 자료 기반 단순 Q&A 1회 완성 *(스트리밍·퀴즈 없이)*

> 목표: "로그인 → 자료 올림 → 질문하면 답이 온다"가 한 줄로 작동하는 상태

### 2단계 — 핵심 완성
- PDF 텍스트 추출
- WebSocket 실시간 스트리밍
- 리프레시 토큰
- 자료 처리 상태 표시
- 근거 자료 위치 표시

### 3단계 — 확장
- 퀴즈 자동 생성 · 채점 · 오답 기록
- 시맨틱 캐싱
- Rate limiting
- 대화 기록 저장
- 소셜 로그인 (구글 → 카카오)

---

## 5. 범위 밖 (이번 MVP에서 다루지 않음)

다음 기능은 의도적으로 제외한다. 향후 여유가 있을 때 확장 후보로 둔다.

- 비밀번호 찾기 / 이메일 인증
- 푸시·이메일 알림
- 다국어 지원
- 결제 기능
- 관리자 통계 대시보드
- 자료 공유 (다른 사용자와)

## 6. 용어 정리

| 용어 | 설명 |
|------|------|
| RAG | Retrieval-Augmented Generation. AI에게 바로 묻지 않고, 관련 자료를 먼저 검색해 함께 전달하여 자료 기반 답변을 만드는 방식 |
| 임베딩 | 텍스트를 숫자 벡터로 변환하는 것. 의미가 비슷한 텍스트는 비슷한 벡터가 되어, 유사도 검색이 가능해진다 |
| JWT | JSON Web Token. 서버가 발급하는 인증 토큰. 서버가 상태를 저장하지 않아도 되는 방식 |
| 리프레시 토큰 | 액세스 토큰이 만료됐을 때 새 토큰을 발급받기 위한 별도 토큰 |
| WebSocket | 서버와 클라이언트가 실시간 양방향으로 데이터를 주고받는 통신 방식. 답변을 한 글자씩 흘려보낼 때 사용 |
| Rate limiting | 일정 시간당 요청 횟수를 제한하는 것 |
